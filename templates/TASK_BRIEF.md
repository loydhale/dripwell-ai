# TASK_BRIEF.md — Template

CTO fills this out and hands it to the Coder for each task.

```
TASK_ID: TASK-<number>
TITLE: <short title>
PARENT_REQUEST: <link or quote of the original Owner request this came from>

GOAL (one sentence):
<what this task accomplishes>

SCOPE:
In scope:
- <bullet>
- <bullet>

Out of scope (do NOT do these even if tempting):
- <bullet>
- <bullet>

FILES LIKELY INVOLVED:
- <path> — <what to do there>
- <path> — <what to do there>

RELEVANT MEMORY ENTRIES:
- LESSONS: <L-numbers if any apply>
- PATTERNS: <P-numbers to follow>
- GOTCHAS: <G-numbers to watch for>

ACCEPTANCE CRITERIA (Auditor will check these):
- <testable criterion 1>
- <testable criterion 2>
- <testable criterion 3>

NOTES / HINTS:
<anything else the Coder needs to know>
```

## Guidelines for CTO when writing briefs

- One goal per task. If you can't write the goal in one sentence, split the task.
- Acceptance criteria must be testable. "Looks good" is not testable. "Returns 200 on valid input, 400 on invalid" is.
- Reference specific memory entries so Coder knows to apply them.
- Out-of-scope bullets prevent scope creep. Always include at least one.
